# Building a portfolio site

## Introduction

So my intention with this was to demonstrate what a 'good' portfolio site should look like for a games programmer or a software engineer 
working within one of the primary programming languages used within the games industry (C/C). I've seen many a portfolio site when hiring junior developers and I've
noticed that many are missing key elements that make them stand out.

I don't feel that there's a need with a portfolio site to have it hosted anywhere else other than on Github as you will no doubt have some of your work on display and
why not have your site located in the same place. GitHub Pages work well and there are many ways to use a static site builder like [Jekyll](https://github.com/jekyll/jekyll) 
or [Hugo](https://gohugo.io/). I've opted to make use of [Markdeep](https://casual-effects.com/markdeep/) which simplifies static site generation by combining markdown which 
I'm quite familiar with and interactive diagrams and styling. Perfect for developers who want a clean, code-first approach to blogging.


Oh and everything I'm coding up is done on a Raspberry pi, as I'd like to demonstrate that you don't need to have an all powerful PC to develop some fun content, plus I enjoy 
the challenge of making everything cross-platfom (not as difficult as it used to be) and I like to code close to the metal for the most part, and outside of a few web games the 
majority of content will be C and C++ which can be compiled quite easily on the Pi. 


I aim to showcase a sensible approach to development, with a focus on performance and analysis to really show why something is engineered the way it is.

## Why Markdeep?

I'm not primarily a web developer and I don't want to spend time setting up complex build tools or learning new frameworks. Markdeep allows me to write my content in markdown, 
which I'm comfortable with, and it takes care of the rest. It's a great way to create a professional-looking portfolio without the overhead of traditional static site generators.


Plus any HTML code I do write is kept to a minimum, and I can focus on the content and the code samples I want to showcase. It's a win-win for me.

## Example Structure

You can find the structure of this site right here on my Github [teacoder.github.io](https://github.com/teacoder/teacoder.github.io). I've structured it as shown, with a few
additional scripts to build a navbar and footer, and a script to populate a list of blog posts from content in the 'posts' directory.

```
this-site/
├── index.html
├── about.md.html
├── projects.md.html
├── posts/
│   ├── 2026-03-02.md.html
│   └── 2026-02-24.md.html
└── style/
│   ├── style.css (optional)
├── scripts/
│   ├── markdeep.min.js
│   ├── posts.js
```

## Key Features

- **Diagrams**: ASCII art rendering with automatic formatting
- **Code Blocks**: Automatic syntax highlighting
- **Responsive**: Mobile-friendly by default
- **Fast**: No server-side processing needed

## Deployment

I've got a couple of scripts that I've written in python to generate post templates, then a script to populate the posts.json file from the content of the posts folder. Once those
have run I just have to push changes to GitHub and then new posts and projects will appear on the site.


Automating the process of creating content is a must for me. I'd like as seamless a process as I can to push content to this site, and this so far is nice and easy without over
complicating it. Plus does allow me to play around a little bit with looking to integrate some WebGL ro Wasm when I get around to it, as I'd like to explore some Emscripten.

## Conclusion

So hopefully I'll begin making some regular posts and showcase a few projects that aspiring game developers or junior engineers can look through to pick up some knowledge and to
find some inspiration for their own portfolio sites.


I don't intend to make this site too discoverable, I'll certaily not be plugging it too hard outside of some 'classroom' demonstrations that I do in my 'day job' for uni students
and secondary visitors to our studio.
